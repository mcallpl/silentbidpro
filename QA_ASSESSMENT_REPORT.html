Fix First — Top 10

Ordered by how badly each one breaks the app for a real donor or admin using it today.

  1. ✓ Fixed Bidding is completely broken. includes/bidding.php:134 used global $mysqli; (never assigned — null), fataling on every bid. Fixing it exposed two more chained bugs underneath: an invalid MYSQLI_TRANS_START_READ_COMMITTED constant and a notification-logging crash against a never-applied schema. All three fixed; bidding now works end-to-end (verified via API and in-browser).
  2. ✓ Fixed Admin auth is fake. The admin gate functions (getAdminFromSession, requireAdminAuth, isAdminLoggedIn) never queried the database — any forged admin_session_token cookie granted full admin access. All three now validate the token against admin_accounts (active accounts only), with a constant-time legacy-token fallback. Verified: forged cookie → 401 / login screen; real session → 200.
  3. ✓ Fixed Live bids can be placed by anyone, unauthenticated. api/bidding/test-bid.php had no auth check — visiting the URL placed a real bid as an arbitrary user. Deleted the file entirely; the URL now 404s.
  4. ✓ Fixed crud-admins.php lets a visitor mint themselves a super-admin account. It never checked super-admin role. Now gated on is_super_admin (403 otherwise), with self-lockout guards. Combined with FIX #2, account management is now fully locked down.
  5. ✓ Fixed Payment webhooks can be spoofed. webhook.php only verified the Stripe signature if a signature header was present. Verification is now mandatory and fail-closed — no signature, no secret, or a bad signature all return 401. Verified with signed/unsigned/tampered payloads.
  6. ✓ Fixed SMS code brute-force protection is dead code. verifyCode() looked up the row by the guessed code, so wrong guesses never counted. Now looked up by phone only, compared with hash_equals(), attempts incremented per wrong guess (lockout at 5, verified), and the code consumed atomically (also fixing a double-consume race).
  7. ✓ Fixed The branding editor saves to a database location the live site never reads. The editor wrote to organizations.brand_* while the CSS reads event_branding/events. Saves now propagate to those per-event tables — verified a color change actually renders on the public page.
  8. ✓ Fixed Admin dashboard list views ignore the event filter entirely. All five endpoints (items/metrics/transactions/users/bids) now enforce event scoping via a shared helper — super admins see all/selected, non-super admins are locked to their assigned events. Verified end-to-end.
  9. ✓ Fixed A winning bidder can pay before the auction even closes. Both checkout.php and create-session.php now require is_closed = 1 (API returns 409 while open). Verified.
  10. ✓ Fixed Every item image in the currently live event is a broken link. Added a branded placeholder + onerror fallback (no more broken icons) and remapped the 5 items with a real thematic match to existing photos. 5 items still need real photography (they show the placeholder).

Feature-by-Feature Status

What a bidder or admin actually experiences today, area by area.

SMS Login / Auth Working

Fixed. Happy path works end-to-end (send → verify → 30-day session). Send-code now reports real delivery failures, OTP brute-force lockout engages at 5 attempts, and codes are consumed atomically (single-use, race-safe).

Placing a Bid Working

Fixed. Bids now place successfully end-to-end — verified via API and in-browser (success toast + "YOU'RE WINNING!" badge). Validation correctly rejects too-low bids without crashing.

Proxy / Max Bidding & Anti-Sniping Working

Now reachable and verified: a max/proxy bid correctly auto-counters a challenger up one increment and retains the lead. The math was always sound; it was simply unreachable until bidding was fixed.

Item Catalog Browsing Working

Fixed. Missing images now show a branded placeholder (no broken icons); 5 items remapped to real photos, 5 await photography. (The "Someone else" false-bidder label is fixed under Bidder-Facing Pages below.)

Checkout / Stripe Payment Hardened

Security fixed: webhook now fail-closed (no spoofing), payment requires a closed auction + real winner, success page checks ownership and shows true status, and failed/expired events resolve. Remaining: a live silentbidpro webhook secret must be added to the vault before go-live.

Auto-Closing Auctions (Cron) Broken

The documented production cron entry calls an admin-auth-gated endpoint from the command line, where it can never authenticate — auctions will never auto-close as documented. A working, unauthenticated script exists but isn't what the README tells operators to schedule.

Legacy Admin Dashboard Auth fixed

The open-door auth bypass is fixed — all admin gates now validate against the DB (forged cookie → login screen). Feature-wise the dashboard renders CRUD/metrics/transactions; remaining work is cross-tenant event scoping (see below), not access control.

Multi-Tenant Events / Orgs Broken

The event/org data model and the newer -event- API family are well-built. But the item, bidder, transaction and metrics views used day-to-day don't scope by event at all — full cross-tenant data leakage.

Event Branding System Mostly fixed

Saving branding now actually renders — colors propagate to the per-event tables the CSS reads (verified). Remaining: the public dev-scratch file BRANDING_EXAMPLES.php and Web Push crypto (below).

SMS Outbid / Winner Notifications Partial

Twilio wrapper itself is solid and fails gracefully. But it's called from a notification-logging path that shares the same "null $mysqli" defect as bidding — a DB hiccup here can crash the very bid-placement request that triggered it.

Web Push Notifications Broken (silently)

Subscribe/unsubscribe endpoints are solid, but the encryption implementation isn't real ECDH — push payloads are sent, logged as "sent," and reported successful, but browsers can never actually decrypt or display them.

QR Codes Working

Generated entirely via an external API / in-memory SVG — no filesystem writes, so the local/production permission differences that bit this project before don't apply here.

Item Document / PDF Generation Partial

Produces a print-ready HTML sheet (not an actual PDF, despite the name), but silently reports success even if the output directory isn't writable, and hard-fails instead of degrading gracefully if the link-shortener API key is unset.

Favorites / Watchlist Working

Clean implementation — checks for the migration before use and returns a clear error rather than crashing if it hasn't run.

▶  Bidding & Auction Closing 7 findings · all fixed ✓
✓ FixedEvery bid placement crashes the server (null $mysqli)
includes/bidding.php:133-138 · api/bidding/place-bid.php
function placeBidWithLocking($item_id, $user_id, $bid_amount, $max_bid_amount = null) {
    global $mysqli;                                       // ← was null
    $mysqli->begin_transaction(MYSQLI_TRANS_START_READ_COMMITTED);

$mysqli was never assigned anywhere in the codebase — the real DB connection lives in a private static variable inside getDB(). Calling begin_transaction() on null threw a PHP Error (not an Exception, so the surrounding try/catch didn't catch it), returning a blank HTTP 500 to the bidder. Introduced in commit 98bd12c, ironically titled "Fix: Critical production issues in bidding and notification systems."

Fix applied: replaced global $mysqli; with $mysqli = getDB();. Because getDB() is a singleton, the FOR UPDATE row lock and the inner placeBid() queries now share one connection/transaction — so the anti-race-condition design actually functions as intended.
Re-tested live: bid places, high bid updates, too-low bids rejected cleanly, and the browser "Confirm Bid" flow now shows "Your bid was placed successfully."
✓ FixedInvalid transaction constant crashed bids (surfaced under #1)
includes/bidding.php:138

With the null-connection bug fixed, the very next line fataled: MYSQLI_TRANS_START_READ_COMMITTED is not a real mysqli constant — READ COMMITTED is a transaction isolation level, not a begin_transaction() start flag. The previous author invented it. This bug was previously invisible because execution never got past the null-connection crash on the line above.

Fix applied: call $mysqli->begin_transaction() with no flags. The SELECT ... FOR UPDATE that follows is what actually provides the row-level lock preventing concurrent-bid races.
Verified live — proxy/max bidding now correctly auto-counters a challenger and retains the lead.
✓ FixedNotification logging crashed committed bids (surfaced under #1)
includes/event-notifier.php:21-48 · notifications table · sql/migrations/006

Once bids could complete, outbidding another user triggered a fatal after the bid had already committed: logNotificationDelivery() inserts a delivery_status column and writes to a notifications_log table that didn't exist — migration 006 was never applied because it used MariaDB/Postgres-only syntax (ADD COLUMN IF NOT EXISTS, partial CREATE INDEX ... WHERE) that is invalid on MySQL 9. Because mysqli runs in exception mode, this returned a 500 to the bidder even though their bid succeeded.

Fix applied: (1) rewrote migration 006 with valid MySQL syntax and applied it (adds delivery_status + creates notifications_log); (2) wrapped the notification dispatch in place-bid.php in a try/catch(\Throwable) so a notification failure can never again turn a successful bid into an error. Also fixed the same undefined global $mysqli in event-notifier.php.
Re-tested live: full outbid flow returns success JSON, notification row is logged, zero fatals.

▶  Authentication & Sessions 12 findings · 11 fixed · 1 open
✓ FixedAdmin authentication is fully bypassable with a forged cookie
includes/auth.php (getAdminFromSession, requireAdminAuth) · admin-accounts.php & admin-auth.php (isAdminLoggedIn)
function getAdminFromSession() {
    $admin_session_token = $_COOKIE[ADMIN_SESSION_COOKIE_NAME] ?? null;
    if (empty($admin_session_token)) { return false; }
    return ['authenticated' => true];   // ← trusted ANY non-empty cookie
}

The gate never touched the database — any non-empty cookie value authenticated as admin, exposing 15+ admin endpoints (users, transactions, items, metrics, bids) plus the ability to create super-admin accounts.

Fix applied: all three admin gate functions now validate the session token against admin_accounts in the database (admin_session_token = ? AND is_active = 1): getAdminFromSession() / requireAdminAuth() in auth.php (now returning the real admin row), and isAdminLoggedIn() in both admin-accounts.php and admin-auth.php. The legacy ADMIN_TOKEN fallback is retained but now uses constant-time hash_equals(). A subtle secondary bug was caught and fixed during testing: dbGetRow returns null (not false) on no match, so an early !== false check still passed — corrected to a truthy check.
Re-tested live: forged cookie now returns 401 on get-users, crud-items, and crud-admins, and shows the login screen (not the dashboard) in the browser; a real DB-backed session returns 200; an inactive account's token is rejected 401.
✓ FixedSMS code brute-force lockout is dead code
includes/auth.php (verifyCode)
WHERE phone_number = ? AND code = ? AND is_used = 0   // ← filtered by the guess
if (!$record) { return false; }   // wrong guess matched nothing → attempts never ++

The lookup filtered on the guessed code itself, so a wrong guess matched zero rows and returned immediately — attempts never reached the 5-attempt limit. An attacker could script unlimited 6-digit guesses for the full 15-minute code lifetime.

Fix applied: the code is now looked up by phone number only, compared in PHP with hash_equals(), and attempts is incremented on every wrong guess; once it hits MAX_VERIFICATION_ATTEMPTS the code is burned. The correct-code path also now consumes the code atomically (UPDATE ... WHERE id=? AND is_used=0 + affected-rows check), which additionally fixes the TOCTOU double-consume race below.
Re-tested live: 5 wrong guesses drive attempts→5 and lock the code out (correct code then rejected); a fresh code with 2 wrong + 1 correct logs in; reusing a consumed code is rejected.
✓ Fixedcrud-admins.php allows self-service super-admin creation

Only checked isAdminLoggedIn() despite the file's own header claiming it "requires super admin privileges." handleCreateAdmin() let the caller set is_super_admin: 1 on a new account — so any authenticated admin (and, before FIX #2, anyone at all) could mint a permanent super-admin login.

Fix applied: the endpoint now resolves the current admin via getAuthenticatedAdminAccount() and returns 403 unless is_super_admin is set. Also added self-lockout guards so an admin can't deactivate their own account or strip their own super-admin role.

▶  Checkout & Payments 6 findings · all fixed ✓
✓ FixedStripe webhook signature check can be skipped entirely
api/checkout/webhook.php:42-48
if ($signature && (STRIPE_WEBHOOK_SECRET || $event_id)) {   // ← skipped if no header
    if (!verifyStripeSignature($payload, $signature, $event_id)) { die(...); }
}

The verification block was skipped whenever the request had no Stripe-Signature header — which an attacker fully controls by simply not sending it. A crafted POST with a fake checkout.session.completed payload and no signature marked any item "paid" for any user with zero real payment.

Fix applied: verification is now mandatory and fail-closed — if (empty($signature) || !verifyStripeSignature(...)) { 401 }. A missing signature, a missing/misconfigured secret, or a failed check all reject before any processing. (The underlying HMAC/replay/hash_equals() logic was already correct.)

Fix applied: both the API (create-session.php) and the page (checkout.php) now require is_closed = 1 before allowing payment (API returns 409 while the auction is still open).

config.php · api/checkout/webhook.php

Methodology

How this assessment was actually produced — so you know what "tested" means here.

The app was run locally against a full clone of the production database schema and current data (PHP 8.4.7 dev server, local MySQL). Every finding above falls into one of two categories, and most fall into both:

  • Live reproduction — the actual HTTP request was made (via curl or a real Chrome browser session driving the real UI, clicking real buttons), server logs and database state were inspected before and after, and the described behavior was directly observed.
  • Static trace — five independent, parallel code reviews each covering one subsystem (bidding/auction engine, auth/sessions, checkout/payments, admin CRUD/multi-tenant, branding/notifications) read the actual source files end to end, cross-referenced against the live database schema, and reasoned through every branch — not summarized or skimmed.

The repository's own prior *.md test reports (BIDDING_TEST_REPORT.md, FORENSIC_TEST_REPORT.md, BRANDING_VERIFICATION_REPORT.md, and others) were deliberately not used as a source of truth for this assessment — several of the critical findings above (the bidding crash, the admin auth bypass, the branding write/read schema mismatch) directly contradict "PASS" verdicts in those reports.
